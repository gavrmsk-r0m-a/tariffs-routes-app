# PostgreSQL schema draft decisions

## 1. Executive summary

- Created `docs/postgres/schema.postgres.sql` as a PostgreSQL DDL draft derived from the current `app/schema.sql`.
- Runtime code was not changed: no PostgreSQL backend, no `psycopg`, no SQLAlchemy/Alembic, no migration script, and no application switch.
- The current SQLite schema remains the runtime source for the application.
- The draft includes the tables currently declared in `app/schema.sql` plus lightweight-migration/runtime support tables found in `app/db.py` and `app/server.py`, including operational tables, dictionaries/reference tables, HLR usage, `app_settings`, `demo_data_state`, import jobs, Telegram settings, route naming rules, change reasons, and `api_tokens`.
- SQLite data-fix `UPDATE` statements at the end of `app/schema.sql` were intentionally not copied because this stage is DDL-only and must not seed or mutate data.

## 2. Type decisions

### IDs

- SQLite `INTEGER PRIMARY KEY AUTOINCREMENT` became `BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY`.
- `GENERATED BY DEFAULT` is intentional so a future migration can insert existing SQLite IDs explicitly.
- Foreign-key columns referencing identity IDs were converted to `BIGINT` so FK types match PK types.

### Booleans

- SQLite integer flags with `CHECK (... IN (0, 1))` became `BOOLEAN NOT NULL DEFAULT false/true`.
- Current runtime SQL still writes/reads SQLite-style `0`/`1` values. Before a real PostgreSQL switch, backend bindings need either an adapter layer or explicit updates to pass/read PostgreSQL booleans consistently.
- Partial indexes using `= 1` became `IS TRUE`.

### Timestamps

- SQLite `TEXT DEFAULT CURRENT_TIMESTAMP` timestamp columns became `TIMESTAMPTZ DEFAULT now()`.
- Target policy: store timestamps as `TIMESTAMPTZ`.
- Before migration, current SQLite `TEXT` timestamps must be normalized and validated.
- Open decision: agree whether legacy server time should be interpreted as UTC or as local server time before casting.

### Dates

- Date-only fields became `DATE`: `rate_date`, `conversion_rate_date`, `old_conversion_rate_date`, `new_conversion_rate_date`, `price_before_conversion_rate_date`, `price_after_conversion_rate_date`, and similar conversion-rate date columns.
- `usage_date` was treated as date-only.
- `valid_from`/`valid_to` were treated as timestamps because SQLite defaults and usage are timestamp-like.

### Numeric / money

- Prices, fees, and EUR amounts use `NUMERIC(18,6)`.
- FX/conversion rates use `NUMERIC(20,8)`.
- Counters remain integer where the existing schema uses integer counts.
- No `FLOAT`/`DOUBLE` is used for money/rates.

### JSON / JSONB

| field | suggested type | reason | migration risk |
| --- | --- | --- | --- |
| `change_log.old_values` | `JSONB` | Change log payload is intended to be structured old-state data. | Need validation/cast; any legacy plain text will fail. |
| `change_log.new_values` | `JSONB` | Change log payload is intended to be structured new-state data. | Need validation/cast; any legacy plain text will fail. |
| `routing_events.snapshot_json` | `JSONB` | Name and usage indicate a structured event snapshot. | Need validation/cast and fallback plan for malformed rows. |

| `import_jobs.error_report` | `JSONB` | Import errors are expected to be structured diagnostics. | Need confirm no plain-text reports exist. |
| `route_history.old_value` | `TEXT` | Field-level audit value can be scalar/mixed text, not guaranteed JSON. | Review if future querying needs structured values. |
| `route_history.new_value` | `TEXT` | Field-level audit value can be scalar/mixed text, not guaranteed JSON. | Review if future querying needs structured values. |

### Enum-like fields

- Enum-like columns remain `TEXT CHECK (...)`.
- PostgreSQL `ENUM` types were not introduced to keep future migrations and value changes simple.

### Partial indexes

- Boolean partial indexes were converted from SQLite integer predicates to PostgreSQL boolean predicates, for example `WHERE is_current IS TRUE` and `WHERE is_active IS TRUE`.

### Foreign-key behavior

- Existing `ON DELETE CASCADE` was preserved for `user_permissions.user_id` and `provider_change_log_servers.provider_change_log_id`.
- Existing `ON DELETE RESTRICT` was preserved where present.
- No new cascade behavior was added.
- Implicit/no-action behavior was not expanded into new business behavior.

### Named constraints

- Important constraints were named to support future PostgreSQL error mapping.
- Index names generally preserve current SQLite names where they already exist.

## 4. Migration risks

- **ID preservation:** migration must insert explicit IDs from SQLite into PostgreSQL identity columns.
- **Identity reset:** after importing each table, reset identity sequences to at least `MAX(id) + 1`; for example: `SELECT setval(pg_get_serial_sequence('routes', 'id'), COALESCE((SELECT MAX(id) FROM routes), 0) + 1, false);` Adjust per table and verify identity sequence names.
- **Boolean conversion:** SQLite `0`/`1` values need conversion to PostgreSQL `false`/`true`; runtime bindings also need review.
- **Timestamp parsing:** SQLite text timestamps need preflight parsing and timezone policy before casting to `TIMESTAMPTZ`.
- **JSON validation:** fields proposed as `JSONB` need validation/cast checks before data import.
- **Numeric precision:** verify existing numeric strings/values fit `NUMERIC(18,6)` or `NUMERIC(20,8)`.
- **Duplicate preflight:** run checks for all unique constraints and partial unique indexes before import.
- **FK orphan preflight:** check every FK for orphan rows before import, especially history/audit tables.
- **DDL-only artifact:** SQLite data normalization `UPDATE` statements from `app/schema.sql` are not included and should be handled in a future data migration/preflight stage.

## 5. Tables needing review

- `route_phone_number_history.old_values` and `new_values`: may be JSON payloads, but remain `TEXT` until production data is inspected.
- `route_history.old_value`/`new_value` and `phone_number_history.old_value`/`new_value`: remain `TEXT` because values may be scalar text.
- `import_jobs.summary` and `error_report`: drafted as `JSONB`; confirm no legacy plain text exists.
- `routing_events.snapshot_json`: drafted as `JSONB`; validate all stored values.
- `hlr_daily_usage` and `app_settings`: created by lightweight migrations rather than `app/schema.sql`; included in the draft, but final ownership should be decided before migration.
- `demo_data_state`: runtime support table created by seed/demo logic in `app/server.py`; included in the draft so preflight inventory matches runtime. Production seed/demo behavior must be separately gated with `ENABLE_DEMO_DATA` before any PostgreSQL cutover.
- `routing_event_servers.id`: SQLite schema uses `INTEGER PRIMARY KEY` without `AUTOINCREMENT`; PostgreSQL draft uses identity for consistency, but this should be reviewed before final migration.
- `valid_from`/`valid_to`: drafted as `TIMESTAMPTZ`; confirm no date-only semantics are expected.
- Case-insensitive uniqueness: current schema does not declare `COLLATE NOCASE`; if application behavior expects case-insensitive uniqueness for names/codes, add functional indexes such as `lower(name)` in a later decision.
- Dynamic tables: no additional runtime-created tables were found in this stage; if future code creates tables outside `app/schema.sql`, document them before migration.

## 6. Not included in this stage

- No runtime PostgreSQL backend.
- No `psycopg` dependency.
- No SQLAlchemy or Alembic.
- No migration script.
- No `app/schema.sql` modification.
- No repository/server/runtime SQL rewrite.
- No seed/demo data.
- No data transfer.

## 7. Next recommended stage

- Add migration preflight checks for duplicate keys, FK orphans, boolean values, timestamp parseability, numeric precision, and JSON validity.
- Then prepare a migration script draft that imports data with explicit IDs and resets all identity sequences after import.
